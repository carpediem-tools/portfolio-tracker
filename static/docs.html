2. Dashboard tab

Consolidated view of the entire portfolio.

Main KPI

  • Total valuation converted to the currency selected in Options (today's rate).
  • Only positions with a live price AND a configured currency contribute to the total.
  • A position whose valuation is unavailable — because its running stock goes negative at some point in time (more sold than held on a given date) — is left out of the total, exactly like a position without a price.
  • When positions are dropped for a missing currency or FX rate, an ⚠️ Excluded positions indicator shows how many.

Consistency banners

Two independent banners may appear above the charts; they cover different problems and can show at the same time:

  • ⚠️ Missing FX rates — at least one rate needed for a consolidation has not been synced yet. A sync is recommended.
  • ⚠️ Negative stock — at least one Securities or Crypto position sells more than it holds at some point in time (its running stock goes below zero on a given date). This is a chronology problem, distinct from the FX banner: check the buy and sell dates and quantities of the flagged position.

Pie charts

Four breakdown charts, all expressed in the Options currency (converted at today's rate):

  • By asset class — one segment per asset class defined in Options, plus an automatic Cryptos segment grouping all crypto positions and a ? segment for securities that have no class assigned
  • Individual CTO positions
  • Individual Crypto positions
  • By broker (CTO only)

⚠️ If exchange rates have not been synced yet, positions in other currencies will not appear in the pie charts.

Annual trend chart

Change in total valuation year over year, fed by data entered in the History tab.

Each history row has its own entry currency. Each value is converted to the Options currency using the row's stored FX rate — the Frankfurter rate at December 31 of the row's year, or the rate at today's date when the row is the current year (for which December 31 has not yet occurred). Rows whose FX rate is missing or invalid are dropped from the chart. The axis and value labels use the symbol of the Options currency.

Annual snapshot table

Below the chart, a table summarizes, year by year: the Securities total, the Cryptos total, the overall Total, and the change (% vs. the previous year). All amounts are converted to the Options currency using each row's stored FX rate. A cell shows — when the FX rate required for that row is missing or invalid.

Since V2.0, the per-row Invested, Valuation, P&L and Weight are expressed in the Options currency, not the native currency. Invested is converted at each purchase lot's own historical rate (the rate on the lot's date), while Valuation is converted at today's rate — the same rate-matching logic a broker uses for realized results. Only Avg cost and Live price stay in the instrument's native currency.

Creating and editing — everything through popups

Since V2.0, the table is display-only: there are no editable cells. All entry happens in popups:

  • + Add position / clicking a position's name → the position popup (Name, ISIN, Yahoo Ticker, Broker, Class).
  • + Add lot / clicking a purchase row → the lot popup (Date, Quantity, Unit price, Fees).

Nothing is written until you validate the popup; Cancel discards everything.

A lot date is now mandatory. The date drives the whole dated cost-basis model (see below and section 12) — a lot cannot be saved without one.

Ticker entry

The ticker identifies the trading venue on Yahoo Finance. It is validated when the position popup is saved: a ticker with an unrecognized suffix is rejected, an error message lists the accepted suffixes, and nothing is written.

Clearing a ticker resets the currency to —.

Important: when the ticker changes, the live price is invalidated automatically (livePrice reset to null, ⚪ status indicator). The currency is not recalculated from the ticker — it remains as previously set by Yahoo Finance, and will be updated at the next sync.

Currency

The currency field is read-only. It is set by Yahoo Finance at each price sync and reflects the actual trading currency of the instrument on its listed exchange. Before the first sync, it displays —.

Positions with currency — are excluded from Dashboard KPIs and pie charts until a sync has been performed.

See section 8 for a full explanation of currency sources.

Purchase sub-table

Clicking a position expands its purchase lots. Each lot row shows:

  • Date, Qty, Price, Fees
  • Lot total = qty × price + fees
  • Lot avg cost = total / qty
  • FX — the historical rate (native → Options currency) at the lot's date, with a ⚪🟢🟡🔴 status indicator and an ✏️ button for manual entry (same logic as the Sales screens, see section 11).

The 🔄 Sync FX rates button in the toolbar fetches these per-lot historical rates from Frankfurter. They are what let the tracker express Invested in the Options currency at each lot's own rate.

The Total row shows the overall quantity, cumulative fees, total invested (native), the overall native WAC, and — when every lot's FX rate is resolved — the overall cost basis in the Options currency.

Remaining quantity and negative-stock detection

The Qty column reflects the quantity still held (bought minus everything sold in the Sales tab, matched by position). The tracker also tracks the running stock along the timeline (buys add, sales subtract, in date order). If that running stock ever drops below zero — you recorded a sale of more units than were held on that date — the position is flagged Negative stock at <date> and its Valuation, P&L and Weight become unavailable (shown blank) until the buy/sell history is corrected. The same situation raises the Dashboard's Negative stock banner (see section 2).

Selling from a position

The - Sell button in the sub-table header opens the sale popup pre-scoped to that position (hidden when nothing remains to sell). See section 5 for how sales work.

5. Securities Sales / Crypto Sales tabs

Record of completed disposals (partial or full sales).

How a sale is created — from the position

Since V2.0, a sale is created only from its position, via the - Sell button in the Securities/Cryptos purchase sub-table. That opens the sale popup, where the asset's identity is fixed and shown read-only: Name, ISIN, Ticker and Currency are copied from the position — you never retype or re-validate them. You then enter only the sell side.

The sale is decoupled from the position afterward: it keeps its own copy of the identity, so deleting the original position does not erase the sale (it is then marked Deleted position). But its cost basis stays linked: as long as the position exists, the cost basis is read back from its purchase lots at display time (see below).

The cost basis is no longer frozen at the moment of sale. Earlier versions stored a fixed "buy average cost" on each sale. Since V2.0 the Avg cost column is recomputed at every display from the position's current purchase lots — specifically, the average cost of the lots dated on or before the sell date. Add a forgotten earlier lot and the figure corrects itself; there is nothing to re-enter. Because a sale no longer carries its own buy side, there is only one FX rate per sale (the sell-date rate). The buy-side conversion now lives on the purchase lots themselves (their per-lot FX rate, section 3).

Temporal quantity cap

The quantity you can sell on a given date is capped so that the position's running stock never goes negative on or after that date. Selling 5 units on a date when only 3 were held is rejected, even if 10 were bought later — you cannot sell in the past units you only acquired afterward. The message states how many were available at that date. When the sell date is after every buy and sale, this reduces to the plain remaining quantity.

Currency and identity are read-only

A sale inherits its currency from its position and never exposes a currency dropdown or an editable ticker — for both Securities and Crypto sales. There is therefore nothing to invalidate on a currency or ticker change at the sale level; the only automatic FX invalidation on a sale is when its sell date changes (see section 11).

Sales KPIs

Four KPIs in the Options currency: Realized P&L, Total B, Total S and P&L %. They share exactly the same perimeter — only sales whose sell-date FX rate is resolved and whose dated cost basis is available. See section 11 for full FX rate management.

  • Sales missing an FX rate or a cost basis (⚪ status indicator) are excluded from the totals; once an FX sync has been attempted, their count is shown below the KPI in gray.
  • After an FX sync has been attempted, a ⚠️ No FX rate KPI appears if at least one sale is still missing a rate, prompting a re-sync.

FX S column

The FX cell carries a ⚪🟢🟡🔴 status indicator. Manual entry of the sell-date rate is done inside the sale popup, via the FX rate (manual) field (available in edit mode) — useful when Frankfurter is unavailable or the broker's exact rate is preferred. See section 11.

FX column

The 🔄 Sync FX rates button in the toolbar fetches or refreshes the Dec 31 rate from Frankfurter for all ⚪ or 🔴 rows. This rate is used by the Dashboard to convert each year's total to the Options currency.

Current year: if December 31 of the current year has not yet passed, the Frankfurter API cannot return a rate for that date. A ⚠️ Dec 31 not yet available — using today's rate notice is shown for that row.

Status indicators follow the same logic as the Sales screens (see section 11).

Entry — through a popup

Since V2.0 the table is display-only. + Add year and the row ✏️ button open the year popup (Year, Currency, Securities, Cryptos, and — in edit mode — a FX rate (manual) field). Nothing is written until you validate; Cancel discards everything. The year is checked on save: it must be a real year, not in the future, and not already present.

When a row is created, the default currency is the current Options currency, and the default year is the next one after the highest already recorded. Both can be changed.

Only the Securities and Cryptos amounts are entered, in each row's own currency. The Total is computed automatically as their sum and is not editable. Manual entry of the FX rate happens inside the popup (the FX rate (manual) field), no longer as a separate inline prompt. On the Dashboard, each amount is converted to the Options currency using the row's stored FX rate (the Frankfurter rate at December 31 of the row's year, or the rate at today's date for the current year). A row whose FX rate is missing or invalid is excluded from the Dashboard chart and snapshot table.

This data feeds the line chart and annual snapshot table in the Dashboard.

Changing the Options currency updates all consolidated displays (Dashboard, top banners, Sales KPIs). It also invalidates the fixed FX rates for all sales (Securities Sales and Crypto Sales): a warning toast is displayed and rates switch to 🔴 (see section 11). 🟡 (manual) rates are also invalidated (they encoded a rate toward the previous Options currency and would produce incorrect results if kept). Outside of sales, it invalidates no positions, triggers no live price re-sync, and pre-fills no fields.

Brokers and asset classes

Configurable lists. Brokers and asset classes can be added and deleted; renaming is not available (an entry is removed and, if needed, re-created under a new name). Each list holds at most 10 entries, and each name is limited to 15 characters.

Deleting an entry that is referenced by one or more open Securities positions is not blocked: a confirmation dialog reports how many positions use it and warns that the corresponding field will be cleared in those positions. On confirmation, the entry is removed and the field is emptied. Only open Securities positions are checked — sales and History rows are not.

Broker and asset-class additions and deletions are saved immediately. By contrast, a change to the reporting currency takes effect only after clicking 💾 Save.

Export / Import JSON

  • Export: downloads portfolio_data.json — useful for backup or migration.
  • Import: loads an existing JSON file and overwrites the current data.

Export CSV (ZIP)

Clicking Export generates an export_YYYYMMDD.zip file containing 6 timestamped CSV files. See section 15 for full details.

8. Currencies — complete guide

This is the most important concept to understand when using the tracker.

Core principle

Seven currencies are supported: EUR, USD, CHF, GBP (£), JPY (¥), HKD (HK$) and CNY (CN¥).

Note on JPY: Japanese yen amounts are displayed without decimal places (e.g. ¥7,203, not ¥7,203.00), consistent with standard JPY notation.

Amount typeCurrencyConversion
Purchase price, Avg cost (WAC), Live priceNative currency (= position currency)Never converted
Purchase-lot detail (price, fees, lot total)Native currencyNever converted
Per-row Invested (Securities/Crypto)Options currencyConverted per lot at each lot's historical rate
Per-row Valuation, P&L, Weight (Securities/Crypto)Options currencyValuation at today's rate; invested side at historical rates
Values entered in HistoryEach row's native currencyNever converted at entry
KPI totals (Dashboard, Securities/Crypto tab banners)Options currencyValuation at today's rate, invested at historical lot rates
Annual trend chart + snapshot table (Dashboard)Options currencyConverted using each row's stored FX rate (Frankfurter at Dec 31, or today's rate for the current year)
Total S (Sales)Options currencyAt the sell-date FX rate
Total B / Realized P&L (Sales)Options currencyCost basis at the buy lots' historical rates, sale proceeds at the sell-date rate

How the invested amount is converted.
Converting a historical purchase at today's exchange rate would be inaccurate — the EUR/USD rate in January 2023 is not the one in 2026. Since V2.0 the tracker instead converts each purchase lot at the rate that applied on its own date (the lot's historical FX rate). Valuation, being about the value now, is converted at today's rate. This is the same rate-matching a broker uses.

Currency sources by context

The currency of an instrument depends on its type and screen:

ContextCurrency source
CTO position (Securities tab)Read from Yahoo Finance (fast_info.currency) at each price sync — — before first sync
Crypto position (Cryptos tab)Inferred from the composite ticker format id:currency (e.g. bitcoin:usd → USD)
CTO Sale (Securities Sales tab)Inherited from the position of origin at creation — read-only on the sale
Crypto Sale (Crypto Sales tab)Inherited from the position of origin at creation — read-only on the sale

Why does the CTO currency come from Yahoo Finance, not the ticker suffix?
A ticker suffix identifies a trading venue, not a currency. The same suffix can cover instruments listed in different currencies on the same exchange. For example, the London Stock Exchange (.L) lists securities in GBP, GBp (pence = GBP/100) and USD. Only Yahoo Finance can authoritatively report the actual trading currency of an instrument. The tracker reads and stores it at each sync.

Impact of changing a CTO ticker

When the ticker of a CTO position is changed:

  • The live price is invalidated (livePrice → null, ⚪ status indicator). A re-sync is needed.
  • The currency is not recalculated from the new ticker — it retains its current value until the next sync, at which point Yahoo Finance will set it to the correct currency for the new ticker.
  • Stored purchases (purchases[]) remain unchanged.

Impact of changing a Crypto ticker

When the ticker of a Crypto position is changed (a sale has no editable ticker):

  • The currency is immediately recalculated from the new composite ticker.
  • If the new ticker is valid, the live price is invalidated (livePrice → null, ⚪ status indicator). A re-sync is needed.
  • Stored purchases (purchases[]) remain unchanged but are reinterpreted in the new currency. For example, a purchase of 15,000 entered when the position was in USD will then display as 15,000 EUR if the ticker is switched to bitcoin:eur. Their per-lot FX rates, which target a specific currency pair, are invalidated and need re-syncing.

Exchange rates

The tracker uses two distinct FX rate sources depending on the use case:

Daily rates — fetched from Yahoo Finance at each synchronization (🔄 button):

  • EURUSD=X : EUR/USD rate
  • EURCHF=X : EUR/CHF rate
  • USDCHF=X : USD/CHF rate
  • EURGBP=X : EUR/GBP rate
  • EURJPY=X : EUR/JPY rate
  • EURHKD=X : EUR/HKD rate
  • EURCNY=X : EUR/CNY rate

Stored in portfolio_data.json under the fxRates key, used for Dashboard consolidations and CTO/Crypto top banners.

Historical rates — fetched from the Frankfurter API (ECB), fixed at a specific date. Used for purchase-lot conversion (Invested), realized P&L in the Sales screens, and the History tab's annual totals. See section 11 for full details.

Ticker Crypto

PartRule

  • ✅ bitcoin:usd, ethereum:eur, solana:chf, cardano:gbp, bitcoin:jpy, ethereum:hkd, solana:cny
  • ❌ BTC:USD, ETH:EUR, SOL:USD (symbols are not recognized by the CoinGecko API)

Where to find the CoinGecko id?
On coingecko.com, the id appears in the URL of the coin's page. For example, coingecko.com/en/coins/bitcoin → the id is bitcoin. The symbol BTC displayed next to the name is different and does not work with the API.

Entry and rejection

CTO tickers (live screens and Securities Sales):

  • If the ticker is empty → accepted, currency and live price reset to —.
  • If the ticker is valid (known suffix or no suffix) → accepted, live price invalidated on live screens. Currency unchanged — will be updated at next sync.
  • If the ticker is invalid (unrecognized suffix) → rejected, an error toast at the bottom of the screen lists the expected suffixes, and the previous value is restored.

Crypto tickers (live screens and Crypto Sales):

  • If the ticker is empty → accepted, currency reset to —.
  • If the ticker is valid → accepted, currency recomputed from the composite format, live price invalidated on live screens.
  • If the ticker is invalid → rejected, an error toast lists the expected format, and the previous value is restored.

10. Yahoo Finance API

This section covers everything related to Yahoo Finance: the Python library, ticker formats, live price states, and troubleshooting.

Library and installation

The tracker uses the Python yfinance library to fetch CTO prices.

pip install yfinance --break-system-packages

yfinance queries Yahoo Finance, a free source requiring no API key. Prices are fetched only when the sync button is clicked — never automatically.

Without yfinance, the tracker falls back to a direct Yahoo Finance API call (less reliable).

GBp normalization: some instruments listed on the London Stock Exchange are quoted in pence (GBp = 1/100 GBP). When Yahoo Finance returns a currency of GBp, the tracker automatically divides the price by 100 and stores the currency as GBP. This conversion is silent — no warning is displayed.

The currency stored on a position is updated at every sync. If the currency changes between two syncs (e.g. after a ticker correction), the new value is stored immediately.

Yahoo Finance troubleshooting

  • Invalid ticker → verify on finance.yahoo.com that the ticker exists and its suffix is in the list above.
  • Full failure (🔴 everywhere) → Yahoo Finance temporarily unavailable, or yfinance not installed (pip install yfinance --break-system-packages).
  • Price missing from Dashboard → a position without a live price (⚪ or 🔴) is excluded from Dashboard KPIs and pie charts.
  • Currency shows — after sync → the fallback HTTP path (without yfinance) does not expose currency information. Install yfinance to enable full currency reading.

11. Historical FX rates (Frankfurter)

Overview

The Frankfurter API is used in three contexts:

  1. Purchase lots (Securities and Cryptos) — a rate fixed at each lot's date, native → Options currency, to express Invested and per-row P&L in the Options currency (see section 3).
  2. Securities Sales and Crypto Sales — a rate fixed at each sale's sell date, to convert the sale proceeds when computing realized P&L.
  3. History tab — the rate at December 31 of each row's year, to convert annual totals to the Options currency in the Dashboard (see section 6).

All three use historical rates fixed at the relevant date, not the current rate. This matches standard broker behavior (Swissquote, etc.): realized P&L in account currency is computed at the rate in effect on each transaction date.

Since V2.0 a sale stores a single rate:

  • fxRateSell: rate (native currency → Options currency) at the sell date.

The buy-side conversion is no longer carried by the sale — it comes from the position's purchase lots, each converted at its own historical rate (context 1 above).

Realized P&L in Options currency = (Sale proceeds × fxRateSell) − (dated cost basis of the sold units, already in Options currency)

Rate source — Frankfurter API

AttributeValue
Sourcehttps://api.frankfurter.dev/ (free service, no API key)
DataOfficial European Central Bank (ECB) rates
CoverageFrom January 4, 1999 (first business day post-euro)
UpdatedOnce per business day, around 4:00 PM CET
CurrenciesEUR, USD, CHF, GBP, JPY, HKD, CNY (all supported currencies)

API response format (example USD → EUR on 2024-01-15):

{"amount": 1.0, "base": "USD", "date": "2024-01-15", "rates": {"EUR": 0.91366}}

Special cases:

  • Weekend or ECB holiday: the API automatically returns the rate from the last business day. No action required. Example: a sale on Saturday May 23 will use the rate from Friday May 22.
  • Date before 1999 or future date: the API returns an error. The row remains without a rate (⚪) — manual entry required.
  • Native currency = Options currency: the rate is automatically 1.0 (no network call, source auto). Example: selling a EUR stock with Options = EUR.

FX rate states (status indicators)

Each sale's sell-date FX rate has a state, shown as a status indicator in the Sales table; purchase-lot rates use the same vocabulary. Manual entry is done inside the sale or lot popup.

IndicatorSourceMeaningRecommended action

When a rate switches to 🔴, the previously calculated value is kept for reference but is no longer used in calculations until re-synced.

Rate synchronization

The 🔄 Sync FX rates button in the toolbar of each Sales screen:

  • Queries the Frankfurter API for all ⚪ or 🔴 rows
  • Overwrites 🟡 (manual) rows: sync is date-based, not source-based
  • Skips 🟢 rows that are already up to date
  • Displays a result toast: FX rates: N OK / M failed

Manual rate entry

For a sale or a History row, the rate is entered inside its popup (the FX rate (manual) field). For a purchase lot, clicking the ✏️ button on the lot's FX cell opens an input box.

Enter the exchange rate (e.g. 1.0742 for 1 USD = 1.0742 EUR).

The source switches to 🟡 manual. This rate will be overwritten by the next FX sync.

Use cases: Frankfurter unavailable, historical rate not available (before 1999), or exact broker rate preferred — in that case, avoid re-syncing after entry.

12. Calculations

This section details the calculation model used by the tracker. Native amounts (purchase price, native WAC, live price) stay in the position's own currency; the invested and valuation figures shown per row and in the KPIs are expressed in the Options currency — invested at each lot's historical rate, valuation at today's rate.

12.1 Position calculations (CTO or Crypto)

Each purchase lot has qty, price, fees, a required date, and a historical fxRate (native → Options currency, at the lot's date). Sales recorded in the Sales tab are matched to the position. From this, the tracker derives:

Total quantity bought (tq)

tq = Σ purchases[i].qty

Sum across all lots. Example: lots of 10 and 5 shares → tq = 15.

Remaining quantity (remaining)

remaining = tq − Σ (linked sales).qty

Quantity still held after every recorded sale of this position. This — not tq — is what drives valuation and invested. Example: 15 bought, 5 sold → remaining = 10.

Running stock and negative-stock detection

Buys (+) and sales (−) are walked in date order (a buy comes before a sale on the same date). The minimum the running stock reaches is tracked: if it ever goes below zero, the position has a negative stock (more sold than held at some point) and its valuation is marked unavailable. Example: sell 6 on a date when only 5 had been bought → running stock hits −1 → flagged.

Native weighted average cost (wac)

wac = ti / tq        where ti = Σ (qty × price + fees)

Fee-inclusive average cost per unit, in native currency. It is invariant to sales (always ti / tq, never ti / remaining). Example: (10×150+5) + (5×160+5) = 2,310 over 15 → wac = 154 €.

Dated cost basis (wacBaseAt(date))

The fee-inclusive weighted average cost of the lots dated on or before date, converted to the Options currency at each lot's own historical fxRate — all-or-nothing: if any relevant lot's rate is unresolved, the result is unavailable (—). Recomputed at every display, never frozen. This is the accessor the Sales screen uses for its Avg cost column (with date = the sell date), replacing the old value that was frozen at the moment of sale.

Invested still held (investedRemaining)

investedRemaining = remaining × wacBase

Cost basis of the quantity still held, in the Options currency. wacBase is the position-level dated cost basis over all lots (all-or-nothing on their FX rates). Unavailable if any lot rate is unresolved or the position has a negative stock.

Valuation (valo)

valo = remaining × livePrice     (native), then converted to the Options currency

Current value of the units still held. Unavailable (excluded from KPIs) if there is no live price or the position has a negative stock. Example: 10 held × 175 € → 1,750 €.

P&L and Chg (gp, evol)

gp   = valuation(Options) − investedRemaining(Options)
evol = gp / investedRemaining

Both in the Options currency: invested is converted at each lot's historical rate, valuation at today's rate. Unavailable if either side is missing.

12.2 Sale calculations (Securities Sales and Crypto Sales)

A sale carries only its sell side (qSold, priceSell, feesSell, sellDate, fxRateSell). Its cost basis is read back from the position's purchase lots at display time — it is no longer stored on the sale. pos is the position of origin (or none, if it was deleted).

Native sale total (ts)

ts = qSold × priceSell − feesSell

Net amount received, in native currency. Example: 5 shares at 200 € with 5 € fees → ts = 995 €.

Dated cost basis (wb)

wb = wacBaseAt(pos, sellDate)          (Options currency, see §12.1)

The fee-inclusive weighted average cost of the lots dated on or before the sell date, converted at each lot's historical rate. null if the position was deleted, has no lot on or before the sell date, or a relevant lot's FX rate is unresolved — the whole sale then shows —.

Buy total, Options currency (tb)

tb = qSold × wb

Cost basis of the sold units, already in the Options currency (no separate buy-side FX rate). Example: 5 × 150.50 € → tb = 752.50 €.

Sale total, Options currency (tsOpt)

tsOpt = ts × fxRateSell

Example (USD→EUR, fxRateSell = 0.92): tsOpt = 995 × 0.92 = 915.40 €.

Realized P&L, Options currency (gpOpt)

gpOpt = tsOpt − tb

Example: gpOpt = 915.40 − 752.50 = +162.90 €.

P&L % (pct)

pct = gpOpt / tb        (in Options currency; null if tb ≤ 0)

Both conditions are required for the Options-currency columns (Total S, P&L, P&L %): the sell-date fxRateSell must be resolved and the dated cost basis wb must be available. If either is missing, those columns show — and the sale is excluded from the consolidated KPIs.

12.3 Currency conversion

The convert(amount, fromCur, toCur) function converts amounts between EUR, USD, CHF, GBP, JPY, HKD and CNY using the rates stored in fxRates. EUR ↔ USD and EUR ↔ CHF use direct Yahoo Finance rates (EURUSD=X, EURCHF=X, USDCHF=X). Conversions involving GBP, JPY, HKD or CNY use EUR-based rates (EURGBP=X, EURJPY=X, EURHKD=X, EURCNY=X); cross-currency conversions not involving EUR (e.g. GBP → USD) triangulate through EUR.

From → ToFormula
EUR → USDamount × eurusd
EUR → CHFamount × eurchf
USD → EURamount / eurusd
USD → CHFamount × usdchf
CHF → EURamount / eurchf
CHF → USDamount / usdchf

GBP, JPY, HKD and CNY conversions all triangulate through EUR. For example: GBP → USD = amount / eurgbp × eurusd; JPY → CHF = amount / eurjpy × eurchf.

If source and destination currencies are the same, the amount is returned unchanged. If the required rate is not available in fxRates (never synced), the conversion returns null and the value is treated as unavailable in downstream calculations.

Example: converting 1,000 USD to EUR with eurusd = 1.08 → 1,000 / 1.08 ≈ 925.93 €.

12.4 Consolidated totals (Securities / Cryptos tabs) and Dashboard valuation

Two distinct consolidations exist, displayed in different places:

  • The Dashboard shows a single consolidated KPI: the total valuation across all live positions (Securities + Cryptos), plus an indicator counting positions excluded for lack of a price or FX rate. It does not display a consolidated invested amount or a consolidated P&L.
  • The Securities and Cryptos tabs each show three KPIs — Invested, Valuation and P&L — consolidated over that tab's own positions, all in the Options currency.

All three consolidations use the same conversion logic described below.

Total valuation (main Dashboard KPI and per-tab Valuation KPI)

Valo totale = Σ convert(position.valo, position.currency, deviseOptions)   (today's rate)

The valuations of all live positions (remaining × livePrice) are summed after individual conversion to the Options currency at today's rate. Positions without a livePrice, with missing FX data, or with a negative stock are silently excluded; an excludedCount indicator appears in the Dashboard.

Example, Options currency = EUR, two positions:

  • AAPL: valo = 2,000 USD, converted with eurusd = 1.08 → 1,851.85 €
  • CW8.PA: valo = 1,500 €, already in EUR → 1,500 €
  • Total Dashboard valuation = 3,351.85 €

Consolidated total invested (per-tab Invested KPI)

Investi total = Σ position.investedRemaining

Sum of each position's investedRemaining, which is already in the Options currency — converted at each purchase lot's own historical rate (§12.1), not re-converted at today's rate. Only positions that have a live price and a fully resolved cost basis are counted. This consolidation appears as the Invested KPI in the Securities and Cryptos tabs (not on the Dashboard) and is used to derive each tab's P&L.

Historical, per-lot conversion. Unlike valuation (today's rate), the invested amount is converted lot by lot at the rate that applied on each lot's date. This is why a position's per-lot FX rates must be synced (section 3) for its Invested — and therefore its P&L — to appear. A position with any unresolved lot rate is left out of the Invested and P&L KPIs.

Consolidated P&L and percentage (per-tab P&L KPI)

G/P consolidé = Valo totale − Investi total
%             = G/P consolidé / Investi total

Calculated from the values already converted to the Options currency, and displayed as the P&L KPI in the Securities and Cryptos tabs.

12.5 Dashboard pie charts

The four pie charts (by asset class, by broker, CTO positions, Crypto positions) use valuations converted to the Options currency, exactly like the main KPI. Positions with unavailable FX data are excluded. No specific calculation: these are views of the same aggregated value grouped by different keys.

12.6 Annual trend chart

Pour chaque année h ∈ historique :
  point.y = h.total × h.fxRate     (h.fxRate = taux ligne → devise Options)

Each history row is converted from its own currency to the Options currency using the row's stored fxRate: the Frankfurter rate at December 31 of the row's year, or — when the row is the current year, for which December 31 has not yet occurred — the rate at today's date. The matching fxRateSource records how the rate was obtained, using History's own vocabulary: frankfurter (rate at Dec 31), today (current-year rate at today's date), auto (row currency = Options currency, rate = 1.0), manual (manually entered), or ko/null when unavailable. Rows whose rate is missing or invalid (ko/null) are excluded from the chart and the snapshot table.

The annual change in the table below the chart is calculated after conversion, i.e. in the Options currency:

Variation année N = (h[N].total_converti − h[N−1].total_converti) / h[N−1].total_converti

12.7 Realized P&L KPI (Sales)

The realized P&L is displayed in the Options currency, consolidated across all sales with available fixed FX rates:

gpTotal = Σ gpOpt   for sales whose sell-date fxRateSell is resolved AND whose dated cost basis is available

Sales missing either (⚪ status indicator) are excluded from this total and counted separately (shown below the KPI in gray). A ⚠️ No FX rate KPI lists the number of incomplete sales. The Total B, Total S and P&L % KPIs share exactly the same perimeter.

See section 11 for complete management of fixed FX rates.

🔄 Sync all prices button (header)

  2. Crypto price fetch via CoinGecko (grouped by currency)
  3. FX rate fetch via Yahoo Finance

Per-tab 🔄 buttons

  • Sync Crypto prices: CoinGecko only + FX

🔄 Sync FX rates button (Sales screens)

Dedicated to historical FX rates for sales. Queries Frankfurter for all rows without a valid rate. See section 11.

Sync result toast

Displays for each type:

  • Stocks: N ✅ M ❌ with the list of failed tickers in parentheses.
  • Crypto: N ✅ M ❌ with the list of failed composite tickers.

A total failure on cryptos (0 ✅) usually indicates an identifier issue: use the CoinGecko id, not the symbol (see section 9).

Prerequisites for CTO prices

pip install yfinance --break-system-packages

Without yfinance, the tracker falls back to a direct Yahoo Finance API call (less reliable). Note: the fallback path does not expose currency information — install yfinance to enable automatic currency reading at sync.

Manual price entry ✏️

Each position has an ✏️ button for manual price entry. Useful for unlisted assets or when synchronization fails. A manually entered price is flagged with the 🟡 indicator (manual source) and is used in calculations like any other price.

Warning: a manually entered price will be overwritten by the next sync (🔄 button). If Yahoo Finance or CoinGecko return a price for that ticker, it will replace the entered value.

Behavior after a ticker change

The live price is invalidated. No automatic sync is triggered at this point: a sync must be launched explicitly. This avoids unnecessary network calls during successive keystrokes (corrections, hesitations).

For CTO positions, the currency is not recalculated at ticker change — it is updated only at the next sync from Yahoo Finance.
